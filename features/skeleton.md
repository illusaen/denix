# `modules/den` and `modules/flake` Skeleton

This document explains the local skeleton under `modules/den` and
`modules/flake`, with emphasis on schema shape and data flow. The root
important detail is that the repository is a `flake-parts` flake whose module
set is loaded through `import-tree`:

```nix
inputs: inputs.flake-parts.lib.mkFlake { inherit inputs; } (inputs.import-tree ./modules)
```

`flake.nix` is generated by `flake-file`; `outputs.nix` preserves the actual
entrypoint above so regeneration does not collapse module discovery into a
single hand-written import list.

## High-Level Shape

The skeleton splits responsibilities into two layers:

1. `modules/flake`: flake mechanics.
   This layer declares inputs, flake-parts integrations, formatter and hook
   tooling, package overlays, devshell contents, diagram generation, and
   generated `flake.nix` settings.

2. `modules/den`: fleet semantics.
   This layer defines den schemas, entity topology, registries, access policy,
   host/user/environment resolution, class routing, quirk declarations, Colmena
   deployment assembly, and defaults that apply to all generated host/user
   systems. It also builds side graphs for ACL and settings lookups with
   `scope-engine`.

Actual instance data is outside these two directories, mostly in
`modules/data`. For example:

- environments are declared in `den.environments`;
- hosts are declared in `den.hosts.<system>.<host>`;
- users are declared in `den.users.registry`;
- access groups are declared in `den.groups`;
- host and role aspects are declared under `den.aspects`.

The `modules/den` directory defines how those data records are interpreted.

## Root Evaluation Pipeline

The root pipeline is:

1. `flake.nix` calls `inputs.flake-parts.lib.mkFlake`.
2. `import-tree ./modules` loads the module tree.
3. `modules/flake/*` adds flake inputs, flake-parts modules, per-system
   outputs, development tools, overlays, and generated-file settings.
4. `modules/den/*` imports upstream `inputs.den.flakeModule`, defines local
   schemas, and wires den policy includes/excludes.
5. Data modules such as `modules/data/environments/*.nix`,
   `modules/data/hosts/*.nix`, and `modules/data/users/*.nix` populate the
   registries.
6. Scope-engine modules evaluate side graphs for ACL and settings lookups from
   the same registries.
7. Den schema and policy resolution turns those registries into scope entities:
   `flake -> fleet -> environment -> host -> user`.
8. Den instantiation policies emit concrete flake outputs such as
   `nixosConfigurations`, `darwinConfigurations`, Home/hjem modules, Colmena
   metadata, devshell routes, wrapper modules, and diagram packages.

## `modules/flake`

`modules/flake` is the repository's flake-parts infrastructure layer.

### `flake-file.nix`

`modules/flake/flake-file.nix` imports
`inputs.flake-file.flakeModules.dendritic` when available and configures the
generated flake file.

It sets:

- `systems = builtins.attrNames den.hosts`; the per-system matrix follows the
  host registry systems.
- `flake-file.prune-lock.enable = true`.
- `flake-file.nixConfig` for substituters, trusted keys, experimental
  features, lazy trees, and XDG store behavior.
- base inputs:
  - `den`
  - `flake-file`
  - `import-tree`
  - `flake-parts`
  - `flake-compat`
  - `flake-utils`
  - `files`
  - `gen-algebra`
  - `nixpkgs-master`
  - `nixpkgs-unstable`
  - `darwin`
  - `hjem`
  - `scope-engine`
- `flake-file.outputs = builtins.readFile ../../outputs.nix`.

That final `outputs` setting is critical. It keeps the generated `flake.nix`
using the local `outputs.nix` entrypoint, which in turn keeps `import-tree
./modules` as the module loader.

### `packages.nix`

`modules/flake/packages.nix` defines the local package overlay and package
context.

Data flow:

1. `perSystem` imports `inputs.nixpkgs` for the current `system` with
   `allowUnfree = true`.
2. It extends that package set with `self.overlays.default`.
3. It assigns the extended package set to `_module.args.pkgs`, so downstream
   per-system modules see the overlaid `pkgs`.
4. It sets `wrappers.pkgs = extended` because wrapper outputs are evaluated
   outside the NixOS module graph and need their own nixpkgs config.
5. It exports selected packages, currently `bambu-studio`.

The overlay itself exposes `config.packages` under `pkgs.local`, using
`withSystem prev.stdenv.hostPlatform.system` to select the matching flake-parts
system.

It also contributes defaults to generated operating-system configurations:

```nix
den.default.nixos.nixpkgs.overlays = [ self.overlays.default ];
den.default.darwin.nixpkgs.overlays = [ self.overlays.default ];
```

### `devshell.nix`

`modules/flake/devshell.nix` adds the `numtide/devshell` input, imports its
flake module, and defines `perSystem.devshells.default`.

The base devshell includes:

- format command from `config.treefmt.build.wrapper`;
- `nh`;
- `nix-tree`;
- `nix-diff`;
- local `custom-scripts` exposed as `nix-build`;
- enabled pre-commit packages;
- `nixd`;
- startup hooks for `load-opnix`;
- a menu-style MOTD when `menu` is available.

The devshell is also a target for den class routing. `modules/den/classes`
defines a `devshell` class so aspects can emit devshell commands/packages and
have them routed into `perSystem.devshells.default`.

### `treefmt.nix`

`modules/flake/treefmt.nix` adds `treefmt-nix`, imports its flake module, and
configures formatters and static checks.

Enabled tools:

- `fish_indent`
- `deadnix`
- `alejandra`
- `statix`
- `shellcheck`
- `toml-sort`
- `ruff` check and format

`treefmt.flakeCheck = false`, so this module provides formatting tooling
without automatically making treefmt a flake check. The devshell imports
`config.treefmt.build.devShell` through `packagesFrom`.

### `pre-commit.nix`

`modules/flake/pre-commit.nix` adds `git-hooks.nix`, imports its flake module,
and configures pre-commit to run the treefmt wrapper.

The pre-commit package is `pkgs.prek`. The default devshell includes
`config.pre-commit.settings.enabledPackages` and runs
`config.pre-commit.installationScript` at startup.

### `flake-root.nix`

`modules/flake/flake-root.nix` adds `flake-root`, sets
`_module.args.rootPath = ../..`, declares `flake-root.projectRootFile =
"flake.nix"`, and adds the flake-root package to the devshell.

The `rootPath` module argument is consumed by other modules/aspects that need a
stable repository root path.

### `diagram.nix`

`modules/flake/diagram.nix` is a documentation artifact pipeline built on
`den-diagram`.

Data flow:

1. Add input `den-diagram`.
2. Collect all hosts from `den.hosts`.
3. Run `den.lib.capture.captureFleet {}` to capture the resolved den fleet
   graph.
4. Project host scopes with `diagram.projectScope { kind = "host"; name = ...;
   }`.
5. Discover user scopes from `fleetCapture.scopeEntityKind` and
   `fleetCapture.scopeParent` instead of reading `host.users`.
6. Render entries for:
   - host graphs;
   - host summaries;
   - user graphs;
   - fleet pipe flow;
   - fleet scope topology;
   - policy resolution map;
   - pipe sequence;
   - fleet DAG;
   - namespace graph;
   - fleet IR JSON;
   - fleet summary.
7. Assemble galleries for fleet, hosts, and users.
8. Expose generated artifacts under `legacyPackages.diagrams`.

The module places diagram packages in `legacyPackages` to avoid forcing the
expensive capture/render path when ordinary tooling walks `packages.*`.

It also creates two helper programs:

- `write-topology`: writes generated `TOPOLOGY.md`;
- `write-diagrams`: writes all diagram entries and galleries.

## `modules/den`

`modules/den` is the local den integration layer. It defines entity schemas,
registry options, policy resolution, class routes, quirks, and fleet defaults.

### `defaults.nix`

`modules/den/defaults.nix` imports the upstream den flake module:

```nix
imports = [ inputs.den.flakeModule ];
```

It then sets `den.default.includes` to shared batteries:

- `den.batteries.inputs'`
- `den.batteries.self'`
- `den.batteries.define-user`
- `den.batteries.primary-user`
- `den.batteries.hostname`

It reserves `settings` as a den structural key:

```nix
den.reservedKeys = [ "settings" ];
```

That prevents aspect-tree traversal from treating `.settings` declarations as
normal nested aspects.

It also sets platform state versions:

- `den.default.nixos.system.stateVersion = "26.05"`
- `den.default.darwin.system.stateVersion = 6`

For users, it appends schema-level includes:

- default user shell via `(den.batteries.user-shell "fish")`;
- a policy named `user-aspect-auto-include`.

`user-aspect-auto-include` checks whether a host-specific user aspect exists at
`den.aspects.${host.name}.${user.name}`. If it exists, it includes that aspect
for the user. This lets host/user-specific customizations be discovered by
convention.

### `fleet.nix`

`modules/den/fleet.nix` defines default fleet user access grants:

```nix
fleet.user-access.by-environment.prod.groups = [
  "system-access"
  "workstation-access"
];

fleet.user-access.by-environment.dev.groups = [
  "system-access"
  "workstation-access"
  "server-access"
];
```

These are not user memberships. They are access grants used later by the fleet
policy to decide which registry users may be resolved onto which hosts.

## Den Schema Model

The local schema is built around these kinds:

- `environment`: an entity;
- `host`: an entity;
- `user`: an entity;
- `group`: data-only registry entries, not part of the den scope tree.

`modules/den/schema/topology.nix` declares parent relationships:

```nix
den.schema.host.parent = "environment";
den.schema.user.parent = "host";
```

Together with the fleet policies, this produces the effective scope tree:

```text
flake
└── fleet
    └── environment
        └── host
            └── user
```

`modules/den/schema/gen-schema.nix` adds input `gen-schema` and imports its
default flake module. The environment registry uses `gen-schema` directly.

Two additional graph evaluators sit beside the den scope tree:

- `modules/den/scope-engine/acl.nix` builds `fleet.acl`, an evaluated ACL
  graph for group inheritance, host/environment gates, and per-host user
  enablement.
- `modules/den/scope-engine/settings.nix` builds `fleet.settings`, an
  evaluated settings graph for root, environment, host, and resolved user
  settings.

These graphs read the same declarations as the den scope walk. They do not
create den entities directly; instead, policies and aspects query them when
they need demand-driven access decisions or settings values.

### Environment Schema

Defined by:

- `modules/den/environments/schema.nix`
- `modules/den/environments/option.nix`

`schema.nix` marks the environment kind as an entity:

```nix
den.schema.environment.isEntity = true;
```

It defines method `getDomainFor`, a schema function that resolves a service
domain:

1. If `services.<name>.domain` is set, return that domain.
2. Else if `services.<name>.delegateTo` is set, return
   `<service>.<delegateTo>.<environment.domain>`.
3. Else return `<service>.<environment.domain>`.

The environment options include:

- `id`: numeric environment ID.
- `domain`: base domain.
- `secretPath`: defaulted to `self + "/.secrets/env/${config.name}"`.
- `wirelessSecretsFile`: defaulted from `secretPath`.
- `settings`: environment-level feature settings for scope cascade.
- `networks`: attrset of network definitions.
- `services`: service domain/delegation definitions.
- `certificates`: certificate domains and issuers.
- `email`: email domain/admin metadata.
- `acme`: ACME server, DNS provider, and DNS resolver.
- `timezone`.
- `location.country` and `location.region`.
- `tags`.
- `delegation.metricsTo`, `delegation.authTo`, `delegation.logsTo`.
- `monitoring.scanEnvironments`.
- `system-access-groups`: environment-level gate groups.
- `access`: map from user names to group lists. This exists in schema, though
  the current user resolution path primarily uses `fleet.user-access`.

`option.nix` builds `options.den.environments` using
`schemaLib.mkInstanceRegistry config.den.genSchema.environment`.

It also adds an `aspect` option to each generated environment instance. That
gives environment entities the same aspect-carrying shape as other den
entities.

### Host Schema

Defined by `modules/den/hosts/schema.nix`.

The host schema marks hosts as entities:

```nix
den.schema.host.isEntity = true;
```

It defines channel metadata:

```nix
channels = {
  nixpkgs-unstable = {
    nixosSystem = inputs.nixpkgs-unstable.lib.nixosSystem;
    darwinSystem = inputs.darwin.lib.darwinSystem;
  };
  nixpkgs-master = {
    nixosSystem = inputs.nixpkgs-master.lib.nixosSystem;
    darwinSystem = inputs.darwin.lib.darwinSystem;
  };
};
```

A validator named `valid-channel` ensures `host.channel` is one of those
channel names.

Important host options:

- `channel`: selects nixpkgs and Darwin/NixOS instantiation function.
- `environment`: environment name; defaults to `"default"`.
- `ipv4`: read-only derived list from the first interface with IPv4 entries,
  stripping CIDR.
- `ipv6`: read-only derived list from the first interface with IPv6 entries.
- `networking.interfaces`: attrset of interface definitions.
- `networking.bonds`: bond definitions.
- `networking.autobridging`.
- `networking.bridges`.
- `system-owner`: primary user for the host.
- `system-access-groups`: host-level gate groups.
- `facts`: defaulted to `self + "/hosts/${config.name}/facter.json"`.
- `secretPath`: defaulted to `self + "/.secrets/hosts/${config.name}"`.
- `public_key`: defaulted to `secretPath + "/ssh_host_ed25519_key.pub"` when
  the secret path exists.
- `exporters`: metrics exporters with `port`, `path`, and `interval`.
- `settings`: typed per-aspect settings namespace.
- `preservation`: persistence settings with `enable`, `disk`,
  `rollbackSnapshot`, and `persistMount`.

The host schema computes `instantiate`:

```nix
instantiate =
  if config.class == "darwin"
  then resolvedChannel.darwinSystem
  else resolvedChannel.nixosSystem;
```

That means host data chooses an instantiation function indirectly through
`channel` plus `class`.

#### Dynamic Host Settings

`host.settings` is generated from the aspect tree. The schema walks
`den.aspects`, skips structural/class/quirk keys, and discovers aspects with a
`.settings` attribute.

The resulting option tree mirrors the aspect namespace. For example, if an
aspect exposes:

```text
den.aspects.disk.zfs-disk-single.settings
```

then the host schema exposes:

```text
host.settings.disk.zfs-disk-single
```

Each discovered settings declaration is reshaped into a submodule with
`imports`, `config`, and options.

## Settings Data Flow

Settings have two related layers:

1. Typed option discovery on host entities.
2. Runtime lookup through the `fleet.settings` scope-engine graph.

### Typed Host Settings

The host schema discovers `den.aspects.*.settings` declarations and exposes a
typed `host.settings` namespace. This is where host data receives validation
and defaults for feature settings.

Data flow:

```text
den.aspects
-> recursively find aspect nodes with .settings
-> skip structural keys, class keys, and quirk keys
-> reshape plain option attrsets into module-shaped settings declarations
-> expose host.settings.<aspect-path>
-> modules/data/hosts/*.nix may set typed values
```

This layer is about schema and validation. It does not resolve inherited
settings by itself.

### Evaluated Settings Cascade

`modules/den/scope-engine/settings.nix` builds `fleet.settings`, a graph for
resolved settings lookup.

Input declarations:

- `den.environments.<env>.settings`
- `den.hosts.<system>.<host>.settings`
- `den.users.registry.<user>.system.settings`
- `den.environments.<env>.delegation`
- user resolution derived from registry groups and host access groups

Graph nodes:

```text
root
env:<environment>
host:<host>
user:<host>:<user>
```

Parent edges:

```text
root -> env:<environment>
env:<environment> -> host:<host>
host:<host> -> user:<host>:<user>
```

Import edges:

```text
env:<environment> -> env:<delegation target>
```

The import edges are derived from:

- `delegation.metricsTo`
- `delegation.authTo`
- `delegation.logsTo`

Resolved user nodes are not every registry user. They are generated for the
users that the host access policy would resolve onto the host.

Settings merge order for `resolvedSettings` is:

```text
parent settings
-> imported settings
-> local settings
```

The implementation uses `engine.shadow`, so more local values shadow imported
and inherited values. The practical precedence is:

```text
user:<host>:<user>
> host:<host>
> imported delegated envs
> env:<environment>
> root
```

The evaluated attributes are:

- `children`: child settings nodes.
- `imports`: imported delegation targets.
- `setting`: parameterized single-key lookup.
- `resolvedSettings`: full merged settings for a node.
- `settingSources`: provenance per resolved key: `local`, `import`, or
  `inherited`.
- `overriddenKeys`: local keys that shadow another value from parent/import
  lookup.

Example lookups:

```nix
config.fleet.settings.get "host:odin" "resolvedSettings"
config.fleet.settings.get "user:odin:wendy" "resolvedSettings"
config.fleet.settings.get "host:odin" "setting" "someFeature"
```

End-to-end settings flow:

```text
aspect .settings declarations
-> host.settings typed option tree
-> modules/data/environments, hosts, users set settings values
-> settings graph builds root/env/host/resolved-user nodes
-> delegation creates import edges
-> resolvedSettings merges parent/import/local data
-> aspects can query fleet.settings for effective values
```

### User Schema and Registry

Defined by:

- `modules/den/user/schema.nix`
- `modules/den/user/option.nix`
- `modules/den/user/assertions.nix`

`schema.nix` defines the reusable user entity shape. It contains:

`identity`:

- `displayName`
- `accountName`
- `email`
- `gpgKey`
- `sshKeys`, each with optional `tag` and required `key`

`system`:

- `uid`
- `gid`
- `linger`
- `enableUnixAccount`
- `extra-features`
- `excluded-features`
- `include-host-features`
- `settings`

`option.nix` defines the concrete registry:

```nix
options.den.users.registry = mkOption {
  type = types.attrsOf registryUserType;
  default = {};
};
```

Each registry user:

- imports `den.schema.user`;
- receives `name` and `userName` from the attrset key;
- defaults `classes = [ "user" ]`;
- defaults `aspect = den.aspects.<name>` when that aspect exists;
- has explicit `groups` used for access matching.

The config section promotes users into real entities:

```nix
den.schema.user.isEntity = true;
den.schema.user.classes = lib.mkDefault [ "hjem" ];
```

`assertions.nix` adds `den.aspects.user-assertions` to user schema includes.
For NixOS users, it asserts that either `password` or `hashedPassword` is set.

`enrichment.nix` adds two user-scope includes:

- `userEnrichment`, which queries `config.fleet.acl.get "host:${host.name}"
  "resolveUser" userName` and emits concrete NixOS account configuration;
- `resolvedUserEmitter`, which emits a compact `resolved-users` pipe record.

The NixOS account enrichment sets:

- `users.mutableUsers = false`;
- `users.groups.${userName}.gid` when `user.system.gid` or `user.system.uid`
  is available;
- `users.users.${userName}.uid` from `user.system.uid`;
- `users.users.${userName}.group` to the private user group, with lower
  priority than explicit user definitions but higher priority than NixOS's
  built-in default;
- `users.users.${userName}.extraGroups` from ACL system groups, excluding
  `wheel` and `networkmanager` because `den.batteries.primary-user` grants
  those separately where needed;
- `linger`, `description`, and `openssh.authorizedKeys.keys` from user
  registry identity/system fields.

The `resolved-users` record contains:

```nix
{
  name = user.name;
  uid = user.system.uid or null;
  groups = user.groups;
  sshKeys = map (key: key.key) (user.identity.sshKeys or []);
}
```

### Group Registry

Defined by:

- `modules/den/groups/option.nix`
- `modules/den/groups/schema.nix`

Groups are intentionally data-only. They do not set `isEntity`, and they are
not resolved into the scope tree.

Each group contains:

- `labels`: for example `posix`, `oauth-grant`, or `user-role`.
- `description`.
- `members`: source groups whose members inherit this group.
- `gid`: POSIX group ID, required by convention for `posix` groups.

`modules/data/groups/*.nix` provides the current group data. For example,
`workstation-access.members = [ "system-access" ]` means members of
`system-access` also inherit `workstation-access`.

## User Access Data Flow

The user resolution path is group-based and policy-driven.

Inputs:

- `den.users.registry.<name>.groups`: groups each user belongs to.
- `den.groups.<group>.members`: group inheritance edges.
- `fleet.user-access.by-environment.<env>.groups`: groups granted to an
  environment.
- `fleet.user-access.by-host.<host>.groups`: groups granted to a host.
- `den.environments.<env>.system-access-groups`: environment-level gate groups.
- `den.hosts.<system>.<host>.system-access-groups`: host-level gate groups.

There are two cooperating implementations:

1. The den policy path decides which user entities are resolved under each host.
2. The scope-engine ACL path computes reusable access records consumed by
   account enrichment and other modules.

### Den User Resolution

1. `resolveUserGroups` expands a user's seed groups through reverse
   inheritance. If group B has `members = [ "A" ]`, members of A inherit B.
2. `resolvedRegistryGroups name` returns all effective groups for a registry
   user.
3. `matchRegistryUsers grantedGroups` selects registry users whose resolved
   groups intersect the granted groups.

Resolution in `modules/den/policies/fleet.nix`:

1. For each environment, read the environment grant from
   `fleet.user-access.by-environment`.
2. For each host in that environment, read the host grant from
   `fleet.user-access.by-host`.
3. Merge environment and host grants.
4. Merge environment and host gates.
5. If gates are empty, use all grants.
6. If gates are not empty, keep only grant groups that are also present in the
   effective gate.
7. Pass the resulting `accessGroups` into the host scope.

Resolution in `modules/den/policies/hosts.nix`:

`den.schema.host.includes = [ den.policies.env-users ];`

This causes `env-users` to fire for every resolved host. `env-users`:

1. receives `host` and `accessGroups`;
2. selects matching registry users;
3. collects user providers from the host aspect and host schema includes;
4. emits `resolve.shared.withIncludes ... { inherit user; }` for each matched
   user.

The result is that users are not manually nested under host declarations.
Users are derived from registry membership plus environment/host access policy.

### Scope-Engine ACL Resolution

`modules/den/scope-engine/acl.nix` builds a separate graph named `fleet.acl`.
It uses `scope-engine` so callers can query ACL values on demand rather than
recomputing group traversal in each aspect.

Graph construction:

1. Flatten all hosts from `den.hosts.<system>.<host>` into one host map.
2. Create parent edges:
   - `root -> env:<environment>`;
   - `env:<environment> -> host:<host>`.
3. Create membership edges named `M`:
   - `group:<member> -> group:<group>` for every `den.groups.<group>.members`
     entry;
   - standalone vertices for all groups.
4. Create declarations for:
   - root;
   - groups, including `scope`, `description`, and `name`;
   - environments, including `system-access-groups`;
   - hosts, including `system-access-groups`.

Group scope is derived from labels:

- `posix` groups become `system` groups;
- `oauth-grant` groups become `kanidm` groups;
- everything else defaults to `system`.

Evaluated ACL attributes:

- `children`: child nodes for a scope.
- `imports`: empty for ACL; the graph is parent/membership based.
- `edges-M`: raw membership edges.
- `effectiveGates`: union of parent environment gates and host gates.
- `resolveUser`: a parameterized lookup.

`resolveUser` is called like this:

```nix
config.fleet.acl.get "host:odin" "resolveUser" "wendy"
```

The result contains:

```nix
{
  userName = "wendy";
  enable = true;
  directGroups = [ ... ];
  allGroups = [ ... ];
  systemGroups = [ ... ];
  kanidmGroups = [ ... ];
  effectiveGates = [ ... ];
}
```

Data flow for `resolveUser`:

```text
den.users.registry.<user>.groups
-> group:<name> IDs
-> follow membership edge M transitively
-> existing group nodes
-> split by group scope
-> read host/environment effectiveGates
-> system group intersection with gates
-> enable boolean + systemGroups/kanidmGroups/allGroups
```

`modules/den/user/enrichment.nix` consumes this record at user scope. The den
policy path decides that `wendy` exists under `odin`; the ACL graph then tells
the account enrichment which POSIX groups and gates apply to `wendy` on
`odin`.

## Fleet Scope Policy Flow

`modules/den/policies/fleet.nix` replaces den's default per-system host walk
with a fleet-first topology.

Policies:

### `to-fleet`

Runs at flake scope and emits one fleet entity:

```text
flake -> fleet
```

The fleet entity is `config.fleet` plus `name = "fleet"`.

### `fleet-to-envs`

Runs at fleet scope and emits one environment entity per
`config.den.environments` entry:

```text
fleet -> environment
```

### `env-to-hosts`

Runs at environment scope and walks all `den.hosts.<system>.<host>` entries.

For each host:

1. Check `hostCfg.environment` against the current environment name.
2. Compute grants and gates as described above.
3. Skip hosts whose `intoAttr == []`.
4. Emit a host entity with `host = hostCfg` and `accessGroups`.
5. Instantiate the host with `den.lib.policy.instantiate hostCfg`.

This produces:

```text
environment -> host
host -> nixosConfigurations/darwinConfigurations/etc.
```

### Include/Exclude Wiring

The fleet policy connects itself like this:

```nix
den.schema.flake.includes = [ den.policies.to-fleet ];
den.schema.fleet.includes = [ den.policies.fleet-to-envs ];
den.schema.environment.includes = [ den.policies.env-to-hosts ];
```

It also excludes upstream defaults that would conflict with this model:

```nix
den.schema.flake-system.excludes = [
  den.policies.system-to-os-outputs
  den.policies.system-to-hm-outputs
];

den.schema.host.excludes = [ den.policies.host-to-users ];
```

So this repository's host and user walk is:

```text
flake
-> fleet
-> environments from den.environments
-> hosts filtered from den.hosts by host.environment
-> users filtered from den.users.registry by accessGroups
```

not:

```text
flake-system
-> all hosts directly
-> host.users directly
```

## Host Aspect and User Provider Flow

Hosts usually have a matching aspect at `den.aspects.<hostName>`. Data files
such as `modules/data/hosts/odin.nix` define:

```nix
den.aspects.odin.includes = with den.aspects; [
  boot
  boot.nvidia
  roles.desktop
];
```

The host schema and den batteries make that aspect part of host resolution.
Then user policy code walks host aspect roots and finds providers for users.

`collectUserProviders` follows:

- `aspect.provides.to-users`;
- `aspect.provides.<userName>`;
- direct `aspect.<userName>` entries unless forwarded;
- explicit `includes`;
- subaspect includes from functor-style aspects.

This is how host-level features can emit user-level configuration. Examples in
the broader module tree include persistence declarations and hjem platform
fragments under `provides.to-users`.

## Class Routing

Den classes let aspects emit abstract module fragments, then policies route
those fragments to concrete output targets depending on scope and platform.

### `classes/devshell.nix`

Declares `den.classes.devshell`.

Routes:

```text
class devshell -> class flake-parts -> path devshells.default
```

This lets aspects emit `devshell = { ... }: { packages = ...; commands = ...;
}` and have it merged into the default devshell.

It also includes `den.policies.system-to-flake-parts` at `flake-system` scope,
which enters the flake-parts scope where devshell routing can happen.

### `classes/env.nix`

Declares `den.classes.env`.

Routes:

```text
class env -> host.class
```

The destination path depends on the platform:

- NixOS: `environment.sessionVariables`
- Darwin: `environment.variables`

The guard only allows `nixos` and `darwin` hosts.

### `classes/hjem-platform.nix`

Declares:

- `den.classes.hjemLinux`
- `den.classes.hjemDarwin`

At user scope:

- if `host.system` ends in `-linux`, route `hjemLinux` into `hjem`;
- if `host.system` ends in `-darwin`, route `hjemDarwin` into `hjem`.

This lets aspects emit platform-specific hjem fragments without forcing every
aspect to know the final user class routing.

### `classes/wrapper-packages.nix`

Imports the wrapper flake module from `nix-wrapper-modules`, adds the
`wrappers` input, exposes `options.wrappers`, and exports wrapper install
modules as `flake.nixosModules`.

It declares `den.classes.wrapper-packages` and routes:

```text
class wrapper-packages -> class flake -> path flake.wrappers
```

The policy is included at host scope, so host/aspect-selected wrappers can
contribute to flake-level wrapper outputs.

## Quirks and Pipes

Quirks are named collection channels used by aspects and policies.

`modules/den/quirks/networking.nix` declares:

- `firewall`
- `host-addrs`

`modules/den/quirks/resolved-users.nix` declares:

- `resolved-users`

`modules/den/quirks/preservation.nix` declares:

- `persistent`
- `cache`
- `persistentHome`
- `cacheHome`

`modules/den/policies/cross-host-quirk-pipes.nix` wires two pipe policies.

`host-addrs` is collected across host scopes:

```nix
den.policies.collect-host-addrs = _: [
  (pipe.from "host-addrs" [
    (pipe.collectAll ({ host, ... }: host != null))
  ])
];
```

Included at host scope, this means every host can collect `host-addrs` pipe
entries from peer host scopes. That supports cross-host features such as
`/etc/hosts` and SSH known_hosts generation.

`resolved-users` is exposed at user scope:

```nix
den.policies.expose-resolved-users = _: [
  (pipe.from "resolved-users" [
    pipe.expose
  ])
];
```

The source is `modules/den/user/enrichment.nix`. Once a user is resolved under
a host, the `resolvedUserEmitter` emits a compact record containing name, UID,
registry groups, and SSH public keys. The expose policy makes that record
available to downstream collectors without requiring them to re-evaluate full
user registry data.

The comment notes that firewall data is not collected here because firewall
configuration is host-specific.

Firewall data follows a local merge path:

```text
aspects emit firewall quirk fragments
-> den.aspects.base.firewall-collector receives firewall
-> lib.mkMerge firewall
-> final NixOS networking.firewall config
```

Preservation data follows class routing rather than cross-host pipes:

```text
aspects emit persist / persistUser class fragments
-> host.preservation.enable gates route policies
-> preservation.preserveAt.${host.preservation.persistMount}
-> system files/directories and per-user files/directories
```

The `boot.preservation` aspect imports `nix-community/preservation`, enables
preservation, preserves core system state, routes user persistence defaults,
and marks the configured `persistMount` as needed for boot.

## Base Networking Flow

`modules/aspects/base/networking.nix` is the main consumer of host and
environment network data.

It emits `host-addrs` at host scope:

```text
host.name
environment.name
environment.domain
host.ipv4
host.public_key
-> host-addrs record
```

Those records are collected back into every host scope by the `host-addrs`
pipe policy. The collected data feeds:

- NixOS `networking.hosts`, excluding the current host;
- OS-level `programs.ssh.knownHosts`, excluding the current host and hosts
  without public key files.

For NixOS interface configuration, the aspect reads:

- `host.networking.interfaces`
- `host.networking.bonds`
- `host.networking.bridges`
- `host.networking.autobridging`
- `environment.networks.default`

The generator classifies interfaces into:

- standalone physical interfaces;
- bridge member interfaces;
- bridge devices;
- bond slave interfaces;
- bond devices.

Data flow for a standalone managed interface:

```text
host.networking.interfaces.<ifname>.ipv4 / ipv6
host.networking.interfaces.<ifname>.dhcp
host.networking.interfaces.<ifname>.managed
host.networking.interfaces.<ifname>.linkLocal
host.networking.interfaces.<ifname>.mtu
host.networking.interfaces.<ifname>.requiredForOnline
environment.networks.default.dnsServers
environment.networks.default.gatewayIp / gatewayIpV6
-> mkManagedNetworkConfig or mkUnmanagedNetworkConfig
-> systemd.network.networks.<ifname>
```

Default behavior:

- if `dhcp` is unset and the interface is managed with IPv4 addresses, DHCP is
  `ipv6`;
- if `dhcp` is unset and the interface is managed without IPv4 addresses, DHCP
  is `yes`;
- unmanaged interfaces default DHCP to `none`;
- managed interfaces default link-local addressing to `ipv6`;
- unmanaged interfaces default link-local addressing to `no`;
- `RequiredForOnline` defaults to `routable`.

Environment DNS is always copied into managed interfaces. Gateway routes are
conditional:

- IPv4 gateway routes are emitted only when the interface has IPv4 addresses
  and those addresses plausibly match the default environment CIDR prefix.
- IPv6 gateway routes are emitted only when the interface has IPv6 addresses.

This matters because the current host data uses `192.168.1.x` addresses while
the declared dev/prod default networks use `10.9.0.0/16` and `10.10.0.0/16`.
The generator can still use environment DNS without installing an obviously
mismatched default IPv4 route.

Bridge flow:

```text
host.networking.autobridging + host.networking.bridges
-> effectiveBridges
-> systemd.network.netdevs.<bridge>
-> systemd.network.links.<bridge>
-> member interface networkConfig.Bridge = bridge
-> bridge device gets IP/DHCP/DNS/routes from its selected interface config
```

If `autobridging = true` and no bridges are declared, each interface is assigned
to an automatically named bridge such as `br0`, `br1`, and so on.

Bond flow:

```text
host.networking.bonds.<bond>.interfaces
host.networking.bonds.<bond>.mode
host.networking.bonds.<bond>.transmitHashPolicy
-> systemd.network.netdevs.<bond>
-> slave interface networkConfig.Bond = bond
-> bond device gets IP/DHCP/DNS/routes from host.networking.interfaces.<bond>
```

Final NixOS output:

```text
networking.useNetworkd = true
networking.useDHCP = false
networking.dhcpcd.enable = false
networking.networkmanager.enable = true
networking.networkmanager.unmanaged = generated networkd interfaces
systemd.network.enable = true
systemd.network.wait-online.enable = false
systemd.network.netdevs = bridges + bonds
systemd.network.links = bridge links
systemd.network.networks = member + standalone + bridge + bond networks
```

On Darwin, the same aspect only sets:

```text
networking.computerName = config.networking.hostName
```

## Colmena Flow

`modules/den/policies/colmena.nix` turns resolved host entities into a Colmena
hive.

Inputs:

- all hosts from `config.den.hosts`;
- host class and system;
- host channel;
- host IPv4 addresses;
- per-host deployment tags from `config.flake.colmenaDeployment`;
- raw host module lists from `config.flake.colmenaModules`;
- `inputs.colmena`;
- nixpkgs inputs for each channel.

Flow:

1. Add Sini's Colmena fork:

   ```nix
   github:sini/colmena/feat/local-system-detection
   ```

   The input follows local `nixpkgs-unstable`, `flake-compat`, and
   `flake-utils`. This fork is used because the hive includes Darwin nodes.
2. Declare `den.classes.colmena`.
3. Define `host-modules-capture`.
4. At host scope, instantiate a synthetic entity named
   `${host.name}-colmena-modules` using the host's own class.
5. Store the raw module list into `flake.colmenaModules.<host>`.
6. Define `host-to-colmena`.
7. At host scope, instantiate a synthetic entity named
   `${host.name}-colmena` with class `colmena`.
8. Flatten class modules into `flake.colmenaDeployment.<host>`.
9. Build `flake.colmenaHive = inputs.colmena.lib.makeHive hiveConfig`.

`hiveConfig` maps each host to a Colmena node:

- imports come from `flake.colmenaModules.<host>`, not from evaluated
  `self.nixosConfigurations` or `self.darwinConfigurations`;
- `deployment.targetHost` is first IPv4 address if available, otherwise
  `<host>.lan`;
- tags include the environment plus aspect-provided Colmena tags;
- `allowLocalDeployment = true`;
- `buildOnTarget` is true when host system differs from local system;
- `targetUser` defaults to `wendy`;
- Darwin hosts get `deployment.systemType = "darwin"`.

The key data-flow improvement is:

```text
host scope module list
-> den.policies.host-modules-capture
-> flake.colmenaModules.<host>
-> hiveConfig.<host>.imports
-> inputs.colmena.lib.makeHive
```

This avoids forcing a complete evaluated NixOS/Darwin configuration just to
recover the module list for Colmena.

The module also adds a Colmena devshell aspect:

```text
den.aspects.devshell.colmena -> devshell class -> devshells.default
```

## Diagram Flow

The diagram pipeline is useful because it visualizes the same scope and policy
pipeline described here.

The important data source is:

```nix
fleetCapture = den.lib.capture.captureFleet {};
```

From there:

- host graphs are `diagram.projectScope { kind = "host"; name = host.name; }`;
- user graphs are discovered from captured scope IDs;
- fleet views are rendered from the capture object;
- graph classes are read from `entity.nodes.*.classes`;
- class-specific views are selected with `rc.views.classViews classes`;
- generated packages are assembled with `entriesToPackages`.

Because user discovery comes from captured scope data, the diagrams reflect the
actual user resolution pipeline rather than raw registry declarations.

## Practical Mental Model

Think of this repository as having three layers:

1. **Declarations**: `modules/data` declares environments, hosts, users,
   groups, roles, and host aspect includes.
2. **Interpretation**: `modules/den` defines schemas and policies that decide
   what those declarations mean, how entities relate, and how access is
   resolved.
3. **Flake packaging**: `modules/flake` turns the interpreted graph into
   developer tooling, generated flake inputs, package overlays, diagrams, and
   concrete flake outputs.

The central design choice is that hosts and users are resolved by a fleet graph,
not by the default per-system den walk. That enables environment-scoped policy,
host gates, group inheritance, cross-host pipes, and generated diagrams to all
operate over one shared topology:

```text
fleet -> environments -> hosts -> users
```
